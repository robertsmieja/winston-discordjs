## 2024-05-18 - Denial of Logging via Discord Embed Limits
**Vulnerability:** Discord API rejects messages with embed field names >256 chars, values >1024 chars, or >25 fields. The Winston transport failed to enforce these limits, meaning maliciously large logs (e.g., from an attacker) would be rejected by Discord and never recorded, hiding traces of the attack.
**Learning:** Logging integrations that rely on external APIs with strict length/size limits must enforce those limits locally (e.g., via truncation) to prevent "Denial of Logging" attacks where an attacker intentionally generates oversized logs to bypass monitoring.
**Prevention:** Always sanitize and truncate log fields before sending them to external APIs with known constraints (like Discord, Slack, etc.).
## 2024-05-18 - Denial of Logging via Unhandled Error Stringification
**Vulnerability:** When a logform field value contains an object without a valid `toString` method (such as `Object.create(null)`), calling `.toString()` to enforce length limits throws a "Cannot convert object to primitive value" or "value.toString is not a function" error, crashing the process and preventing the log from being sent.
**Learning:** Any dynamic values handled by a logging transport must have their type/behavior verified before conversion. Assuming `toString` exists and functions correctly is dangerous and creates a denial-of-logging vulnerability when a malicious payload is supplied.
**Prevention:** Implement robust stringification fallbacks (e.g., using a try/catch block with `JSON.stringify` or returning a default string like `"[object Object]"`) to safely process dynamic object payloads before sending them to the logging provider.
