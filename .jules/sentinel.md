## 2024-05-18 - Denial of Logging via Discord Embed Limits
**Vulnerability:** Discord API rejects messages with embed field names >256 chars, values >1024 chars, or >25 fields. The Winston transport failed to enforce these limits, meaning maliciously large logs (e.g., from an attacker) would be rejected by Discord and never recorded, hiding traces of the attack.
**Learning:** Logging integrations that rely on external APIs with strict length/size limits must enforce those limits locally (e.g., via truncation) to prevent "Denial of Logging" attacks where an attacker intentionally generates oversized logs to bypass monitoring.
**Prevention:** Always sanitize and truncate log fields before sending them to external APIs with known constraints (like Discord, Slack, etc.).

## 2025-03-26 - Denial of Logging via Unsafe Stringification
**Vulnerability:** The logging formatter (`src/LogHandlers.ts`) used implicit `.toString()` stringification via template literals (e.g., `${value}`) or explicit calls like `value.toString()`. If the application logged an object without a valid prototype chain (e.g., `Object.create(null)`) or an object that threw an error within its `toString` method, it would cause an uncaught TypeError or error propagation, completely crashing the logging pipeline and potentially the Node.js process itself, leaving no trace of the log.
**Learning:** Logging logic MUST be extremely robust against malformed or unexpected data structures. Relying on default language coercion or user-provided `toString` methods in a critical logging path can lead to Denial of Service (DoS) or Denial of Logging vulnerabilities if an attacker or buggy code can inject such objects into the logging pipeline.
**Prevention:** Always use safe stringification utilities (like a try-catch wrapped `JSON.stringify` or a verified `toString` check) when handling arbitrary input data in logging integrations or external API boundaries.
