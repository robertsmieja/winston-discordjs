## 2024-03-24 - `toLocaleUpperCase` vs `toUpperCase`
**Learning:** For hardcoded property names, system string values, or log keys (e.g. `timestamp`, `level`, `message`), `toLocaleUpperCase` introduces significant performance overhead and runs >10x slower than `toUpperCase`. V8 optimizes `toUpperCase` more aggressively since it does not have to consider runtime locale resolution. Furthermore, `toUpperCase` is functionally safer for system strings because different system locales might capitalize certain characters differently (e.g. Turkish `i` to `İ`).
**Action:** When capitalizing technical strings and property keys, always use `toUpperCase()` over `toLocaleUpperCase()`. Additionally, always lift small functions like `capitalize` out of the render/logging functions (e.g. extracting it out of `handleLogform`) to avoid costly closure recreation on every event invocation.
